Data Volume

1. 環境準備

1. my_cron 文件

  • 我的專案資料夾中，有一個 my_cron 文件，內容如下：
*/1 * * * * date >> /app/test.txt
  • 所以看得出來，這個文件就在寫 crontab 裡的內容：每分鐘執行一次，會執行 linux 的 date 這個 funtion。並把印出來的時間，存到 /app/test.txt 裡面

2. Dockerfile

  • 我的專案資料夾中，也有一個 Dockerfile，內容如下：
    FROM alpine:latest
RUN apk update
RUN apk --no-cache add curl
ENV SUPERCRONIC_URL=https://github.com/aptible/supercronic/releases/download/v0.1.12/supercronic-linux-amd64 \
    SUPERCRONIC=supercronic-linux-amd64 \
    SUPERCRONIC_SHA1SUM=048b95b48b708983effb2e5c935a1ef8483d9e3e
RUN curl -fsSLO "$SUPERCRONIC_URL" \
    && echo "${SUPERCRONIC_SHA1SUM}  ${SUPERCRONIC}" | sha1sum -c - \
    && chmod +x "$SUPERCRONIC" \
    && mv "$SUPERCRONIC" "/usr/local/bin/${SUPERCRONIC}" \
    && ln -s "/usr/local/bin/${SUPERCRONIC}" /usr/local/bin/supercronic
COPY my-cron /app/my-cron
WORKDIR /app
# RUN cron job
CMD ["/usr/local/bin/supercronic", "/app/my-cron"]

2. 預設作法：只存在container 中的 container layer

  • 那照之前的做法，我們接下來就是
    • 建立 image： docker image build -t my_cron .
    • 開 container 跑他： docker container run my_cron
    • 確定一下這個 container 的 id，： docker container ls
    • 確認這個 container 有正確在工作：
      docker image build -t my_cron . # 建立 image
docker container run my_cron # 開 container 跑他
docker container ls # 確定一下這個 container 的 id，
docker container exec -it {container_id} sh # 用互動模式進入
ls # 目前在 container 的 /app 下面，所以應該會看到 my-cron 和 output.txt
more output.txt # 應該會看到每分鐘的紀錄
    • 暫停這個 container : 
      exit # 離開互動模式
docker container stop {container_id}
    • 恢復這個 container，並確認資料都還在：
      docker container start {container_id}
docker container exec -it {container_id} sh # 用互動模式進入
ls # 目前在 container 的 /app 下面，所以應該會看到 my-cron 和 output.txt
more output.txt # 看到剛剛停止前，每分鐘的紀錄
    • 暫停並刪除這個 container
      exit # 離開互動模式
docker container stop {container_id}
docker container rm {container_id}
    • 重啟 container，資料都不在了
      docker container run my_cron
docker container ls # 看一下 container 的 id，
docker container exec -it {container_id} sh # 用互動模式進入
ls # 目前在 container 的 /app 下面，所以應該會看到 my-cron 和 output.txt
more output.txt # 剛剛的紀錄全沒了。因為現在是在另一個 container 裡面了

    3.1 不指定 -v 參數

    • 我們先改寫一下 Dockerfile，只要加上一行 VOLUME ["app"] 
    FROM alpine:latest
RUN apk update
RUN apk --no-cache add curl
ENV SUPERCRONIC_URL=https://github.com/aptible/supercronic/releases/download/v0.1.12/supercronic-linux-amd64 \
    SUPERCRONIC=supercronic-linux-amd64 \
    SUPERCRONIC_SHA1SUM=048b95b48b708983effb2e5c935a1ef8483d9e3e
RUN curl -fsSLO "$SUPERCRONIC_URL" \
    && echo "${SUPERCRONIC_SHA1SUM}  ${SUPERCRONIC}" | sha1sum -c - \
    && chmod +x "$SUPERCRONIC" \
    && mv "$SUPERCRONIC" "/usr/local/bin/${SUPERCRONIC}" \
    && ln -s "/usr/local/bin/${SUPERCRONIC}" /usr/local/bin/supercronic
COPY my-cron /app/my-cron

# RUN cron job

    3.2 指定 -v 參數

    • 剛剛那樣做，雖然資料都寫在主機上，不會隨著 container 刪除而消失了
    • 但我還是我不太滿意，因為我如果刪除後，重開新的 container，那資料會被存在另一個主機資料夾
    • 我希望資料都存在同一個主機資料夾裡面，我希望這個 output.txt 可以接續的存新資料進去
    • 那作法就是，加上 -v 參數，指定 volume 名稱
    • 首先，Dockerfile 不需要再寫 VOLUME ["/app"] 了，Dockerfile 變回最一開始的樣子：
      FROM alpine:latest
RUN apk update
RUN apk --no-cache add curl
ENV SUPERCRONIC_URL=https://github.com/aptible/supercronic/releases/download/v0.1.12/supercronic-linux-amd64 \
    SUPERCRONIC=supercronic-linux-amd64 \
    SUPERCRONIC_SHA1SUM=048b95b48b708983effb2e5c935a1ef8483d9e3e
RUN curl -fsSLO "$SUPERCRONIC_URL" \
    && echo "${SUPERCRONIC_SHA1SUM}  ${SUPERCRONIC}" | sha1sum -c - \
    && chmod +x "$SUPERCRONIC" \
    && mv "$SUPERCRONIC" "/usr/local/bin/${SUPERCRONIC}" \
    && ln -s "/usr/local/bin/${SUPERCRONIC}" /usr/local/bin/supercronic
COPY my-cron /app/my-cron
WORKDIR /app
# RUN cron job
CMD ["/usr/local/bin/supercronic", "/app/my-cron"]
    • 重新 build image 並 開container
      docker image build -t my-cron .
docker container run -d -v cron-data:/app my-cron
# 43c6d0357b0893861092a752c61ab01bdfa62ea766d01d2fcb8b3ecb6c88b3de
    • 可以看到，加上 -v cron-data:/app ，意思就是，我要把 container 裡面的 /app 資料夾，mount 到我主機的 cron-data 這個資料夾
    • 但... cron-data 這個資料夾在哪？他其實會被預設存放在 var/lib/docker/volumes/ 的底下：
      docker volume ls
# DRIVER    VOLUME NAME
# local     cron-data
docker volume inspect cron-data
# [
#     {
#         "CreatedAt": "2021-06-22T23:25:02+02:00",
#         "Driver": "local",
#         "Labels": null,
#         "Mountpoint": "/var/lib/docker/volumes/cron-data/_data",
#         "Name": "cron-data",
#         "Options": null,
#         "Scope": "local"
#     }
# ]
ls /var/lib/docker/volumes/cron-data/_data
# my-cron
ls /var/lib/docker/volumes/cron-data/_data
# my-cron  test.txt
    • 所以，這和我之前學的 mount 在我指定的資料夾的差別，就在於他是存在 Docker 統一管理的存放位子 var/lib/docker/volumes/ 底下而已。
    • 那如果這個 container 被刪掉後，只要重啟的 container ，也是下 docker container run -d -v cron-data:/app my-cron 這個指令開 container，那他就一樣會把主機的 cron-data 和你 container 中的 /app 資料夾連動，所以就能繼續把資料寫進 output.txt 裡面了
