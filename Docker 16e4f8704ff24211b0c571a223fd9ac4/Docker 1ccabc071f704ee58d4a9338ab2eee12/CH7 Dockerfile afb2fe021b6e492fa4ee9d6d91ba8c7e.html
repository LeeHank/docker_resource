CH7: Dockerfile

7.1 觀念解說

  • 首先看最下面的，MacOS，也就是我們的主機。
  • 接著，左手邊那個block，藍色的docker client，就是我們mac上灌來跑docker的terminal，例如我的筆電是mac，然後我的docker desktop就是一個docker client，我只要開啟docker desktop服務後，我就可以在terminal下docker blabla...的指令。所以當我在terminal上下docker指令時，我就是利用Docker Client這個工作者在做事。
  • 再來，往上看到綠色的build context這層，這是指，當我下docker build .這個指令要去build image時，這時執行指令的這層目錄(i.e. .這個目錄)，會被稱作build context。而這層目錄中，一定會有Dockerfile這個檔案，以及其他我們要使用的檔案(e.g. Other Files)。特別注意，這些檔案都是存在於Mac的硬碟中。
  • 接著看右邊的Linux OS (VM)這塊。這塊是在講，我們在Mac上，用VM架了一個Linux OS，然後在Linux OS上安裝一個docker engine。所有在Docker Client中下的docker指令，都會傳到Docket Engine。所以剛剛我們下了docker build .的指令，也會傳遞到Docker Engine。當Docker Engine接受到docker build .的指令後，就會把左邊的build context(也就是mac上要拿來build image的那個資料夾)，copy一份，到Linux的硬碟中。所以就會看到copied context這一層。這一層就是copy過來的資料夾，裡面就會有Dockerfile和Other Files這些檔案。
  • 在這邊先打住一下。目前我們已經看到兩個作業系統，兩個工作者，以及兩個空間：
    • 作業系統： 左圖是在Mac OS下，但右圖是在Linux OS下
    • 工作者： 左圖的工作者叫Docker Client，右圖的工作者叫Docker Engine
    • 空間： 左圖的檔案(i.e. Dockerfile, Other Files)，是放在Mac硬碟下。右圖的檔案，是放在Linux這個虛擬機的硬碟空間下(還不是container喔，就真的是放在Linux的硬碟下)
  • 另外，講到現在，都是假設我們的筆電是mac，然後我用docker desktop在做事，所看到的架構。但如果今天是在公司，那我其實根本不管筆電是Mac or Windows，我們都是直接進到linux server (e.g. 我們在windows上，會用putty登入linux server，然後就直接在這個server下下指令做事了)。如果是這樣的話，那我會把我要build image的檔案，直接放在linux中的某個資料夾中，所以現在這張圖，我可以把左邊全部砍掉不要看。然後中間的Copied Context，把它改成Build Context即可。
  • 再貼一次圖，再繼續講
  • 接著，我們要看到最右邊了。這邊要開始講Docker Engine如何去執行Dockerfile中的指令。
  • 我們首先看到第一個Dockerfile command，這邊是指，Docker Engine去讀Dockerfile時，第一行會讀到的Command，一定都是FROM ...，這是用來找base image用的。例如這邊的FROM alpine:latest，就是要去取最新版本的alpine這個image(alpine是一種linux作業系統，你就把它想成和centOS一樣的東西，都是linux OS)。Docker engine抓完後，就會建立紅色這層Base image，並開啟一個臨時用的container，我們就叫他tmp Container01，這個container會以這個base image為內容起起來。
  • 再來，Docker Engine就會去讀Dockerfile的第二行指令，例如這邊的RUN touch file001.txt，就是要開一個叫做file001.txt的檔案。
  • 這邊提醒一下，在Dockerfile中，重要的指令(e.g. 這邊的RUN)，都會在原本的container上再加一個layer，也就是這邊的tmp image01這個新layer。簡單來想就是，我本來先引入base image(也就是乾淨的alpine Linux OS)，那現在就是更新image，這個image是alpine Linux OS，加上裡面已經有file001.txt這個檔案的狀態。
  • 那更新完image後，就會再加上一個container，來繼續run dockerfile後面的指令，所以就看到了tmp Container02這個container。
  • 接著run Dockerfile第三行指令，COPY /src /dst。這邊的/src指的是Copied Context，他是存在於linux虛擬機上的硬碟空間。/dst是指tmp Container02，他是Docker Engine正在建立image時，開起來的container空間。所以這邊我們改用imported files這個字。
  • 有了這些檔案後，我就又可以更新我的image了，所以現在更新完的image，我叫做tmp image02
  • 有了新的image，他就又會開啟一個container，然後繼續讀Dockerfile的下一行command。讀到ENTRYPOINT ["httpd"]。ENTRYPOINT通常是dockerfile的最後一行指令了，他在幹麻現在先不介紹，但總之有了這個指令，Docker Engine就又會更新image，而這個image已經是最終版了，所以叫他final image。那就完成image的建置。
  • 總結一下:
    • 有兩個工作者：Docker Client和Docker Engine。我們在自己電腦都是透過Docker Clinet在下docker指令，然後這個指令會傳遞到Docker Engine，Docker Engine就會在Linux虛擬機中，幫我們執行這些指令。
    • 整個過程中有三個空間(想成硬碟就好)：第一個是Mac OS中的空間，我們在這個空間中，放置要建立image所需的檔案(i.e. 最左邊的Dockerfile和Other Files)。這個資料夾我們叫他Build Context。第二個空間是中間的Linux虛擬機的硬碟空間，我們在這個空間中，放置從Mac OS copy過來的資料夾，我們叫copied context，裡面就是放要建立image用的Dockerfile和Other Files。第三個空間是最右邊的，由Docker Engine臨時建起來的container空間(可以想成ram)。這些都是在建立image時臨時開出來的空間，用來放置實際在建立image時要用的檔案，也就是imported files。我們最終就會把這些檔案和指令，包成final image。
