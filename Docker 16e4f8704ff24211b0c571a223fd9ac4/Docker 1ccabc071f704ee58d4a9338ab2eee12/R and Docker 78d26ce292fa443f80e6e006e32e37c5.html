R and Docker

  • 這邊先摘要 RStudio 的這篇文章，言簡意賅： https://environments.rstudio.com/docker

0. 前言

  • Docker是個很大的主題，這篇文章只focus在介紹，如何用Docker，來建立一個 R 的 reproducible environments
  • 有四個重要的概念一定要很清楚：
    • Dockerfiles
    • Images
    • Registries
    • Containers
  • 這四個概念清楚後，就開始把重點放在 Layers in a container

1. 四個重要概念

  • 作者把理解 Docker 的四個概念，拿來和釀造與享用啤酒做類比
    • Dockerfile: 要建立一個環境時，所需要的步驟的筆記。這就像釀啤酒時要看的recipe。
    • Image: 照著Dockerfile去執行這些步驟後，所得到的環境，就叫image。這就像釀啤酒時，你照個recipe去製作，最後就會得到這種recipe下的一批啤酒(e.g. 台啤recipe，做出台啤)
    • Container: 剛剛說的 image ，指的環境，就是 os + linux package + R package。那這個環境要運行在某個硬體上 (e.g. 主機, 虛擬機)，那 container就是架在 docker engine 上的一種硬體，我們打開這個 container 後，就可以運行 image ，來得到我要的環境了。這類比回剛剛的啤酒，就是我們會找個地方(i.e. container)，來打開啤酒好好享用的意思。
    • Registry：放各種 image 的線上平台，我們可以去這些平台下載要用的 image ，也可以把自己的image傳到這些平台。這就像啤酒商店，裡面有各式各樣的啤酒，你想買啥就買啥。
  • 這四個概念可以畫成下圖：
  • R user 平常在 run 的 process 大概分成兩個，這兩個都和 container 息息相關：
    • Development session
    • Production Runtime

2. Example Registries

  • 這邊我想先提一下，我們如果要下載 image 的話，可以去哪些地方下載。
  • 在 R 的使用上，主要有三個 project 在負責。他們的目的不太一樣，所以造出來的 image 也有所不同：
    • Rocker Project：
      • 先上連結：https://www.rocker-project.org/
      • 這個 project 的目的，是去建立一系列可以 self-contained 的 image，來讓 R user 做 R 的應用開發。
      • 所以，我們最常下載他們的 image 來用，因為 hen 貼心。舉例來說， rocker/tidyverse 這個 image ，就幫你包好 OS, system dependency, R, tidyverse, RStudio server 這些。所以一打開他就很完整了。又例如 rocker/shiny-verse 就是幫你包好 shiny server + tidyverse 等等。
      • 提醒一下，他們家的 image，作業系統都是用 Debian OS (不知道為何)
    • RStudio Images：
      • 先上連結：https://github.com/rstudio/r-docker
      • 這個 project 的image，只簡單做到OS+dependency+R verison而已。例如，可用 docker pull rstudio/r-base:4.0-xenial 來下載 ubuntu-xenial這個作業系統搭配R4.0的image。因為這個 project 的目的，是讓使用 RStudio Launcher 的人，有個base layers來用而已。
      • 如果想看細節，可以點進連結，就會發現，啊哈，這不就是我最熟悉的 RStudio 的 github嗎？沒錯，他有一個專案資料夾叫 r-docker ，裡面就是把 R的版本 分門別類為各自的資料夾。然後在給定版本的資料夾，再分各作業系統的資料夾。然後在指定作業系統的資料夾下，就可以看到對應的Dockerfile了。
    • R-Hub:
      • 這個我還沒整理，但他的目的，是for package開發的人用的。
      • 當我們開發完package，傳到CRAN時，CRAN會有一系列的環境，去對你的package做測試。那R-Hub這個專案，就是幫你把這些環境包好成image，那你去下載這些image，開container來run這些image，就可以檢驗你開發的package，是不是在這些環境中運行都可以過關。

3. Layers in a Container

  • 一個 R 的 data science container，會包含這六個基本 components
    1. Base Operating System: 就是我們是跑在哪個OS上
    1. System dependencies
    1. R
    1. R Packages
    1. Code
    1. Data

  • 那這個 container 裡面要跑這麼多事，可以有一部分分工給 image 來做。而這個 image，他最少可以只包含1，最多可以包含1到6
  • 舉例來說
    • 我可以只下載 ubuntu xential 的 image，所以此時我開一個container來run 這個 image 時，我得到一個 ubuntu 的環境。然後，我在此container裡面，就好像進到linux server裡面，我可以依序用 command line，來灌 linux package(就是第2點的system dependencies，例如圖中的 libssl 這個 package)，灌R，灌R packages，用touch開RMarkdown寫code，開文字檔用逗點分格寫data存data，再run我的.Rmd檔。
    • 我也可以下載 rocker/tidyverse 的image，此時這個image已經幫你包好 1~4 (OS他是用 Debian，System dependencies已幫你灌好，R用最新版 4.0.0，R packages 幫你裝好 tidyverse的所有東西)，所以你container一打開去run他，你就已經完成 1~4 了，接著，你就在這個container裡面(其實就是個灌好tidyverse的linux server)，去寫你的code就好
    • 我可以自己寫個 Dockerfile，這個file裡面，我把步驟寫好：先下載 rocker/tidyver image → 再去 github 上 clone 我寫好的 code → 再從本機 copy 我的 data 進來。那我用這個Dockerfile去build出我的專屬image時，他裡面就包含 1~ 6 了。這就是常被提及的 docker 部署包。一打開就可以跑你的app了。
  • 接下來，就依序細講這六個 componet，如果要寫在 Dockerfile裡面怎寫。或是，不寫在 Dockerfile，那他在開啟container後，要如何寫，才能運作好。

3.1 Base OS

  • 這個是一定要用 image 的。所以作法是：
    • 先去DockerHub，下載別人寫好的Base OS image。例如 docker pull ubuntu:xential
    • 寫 Dockerfile，第一行就是寫剛剛下載好的 image。例如 FROM ubuntu:xential
  • 坦白說，這部分我們很少用，因為我們通常都是下載 R 的 image，那這個 image 他已經幫我們從 Base OS, system dependency, 一路包到 R了。

3.2 System Dependencies

  • 安裝R時，需要一些system libraries 才能run。detail可以看這個 連結。但這部分我們不用太擔心，因為我們通常會下載 R 的 Imgae來用，這些 image 已經都幫我們包好 OS 和 system dependency了。
  • 除了R以外，R的package也會需要對應的一些 system libraries。這些system libraries可以從這個package的description file裡面看到，或是，可以用 sysreq 這個 R package幫你自動下載這些系統套件。
  • 對我們來說，處理 R package要用的 system dependency比較常會碰到。因為我們自己寫的dockerfile，可能是從 rocker/tidyverse 開始，然後會繼續安裝我需要的 package。那在裝這些 package時，就可能遇到 error，告訴你沒有安裝需要的 system dependency。這時就要把這些system dependency也寫進Dockerfile裡面。
  • 以下，用官網的例子來看怎麼把 R 的 system dependencies 和 R package 的 system dependencies寫進Dockerfile內。(官網建議要分開寫，比較清楚)
FROM ubuntu:xenial
# Install system dependencies for R
RUN apt-get update -qq && \
    DEBIAN_FRONTEND=noninteractive apt-get install -y \
    apt-transport-https \
    build-essential \
    curl \
    gfortran \
    libatlas-base-dev \
    libbz2-dev \
    libcairo2 \
    libcurl4-openssl-dev \
    libicu-dev \
    liblzma-dev \
    libpango-1.0-0 \
    libpangocairo-1.0-0 \
    libpcre3-dev \
    libtcl8.6 \
    libtiff5 \
    libtk8.6 \
    libx11-6 \
    libxt6 \
    locales \
    tzdata \
    zlib1g-dev
    
# Install system dependencies for the tidyverse R packages
RUN apt-get install -y \
    make
    libcurl4-openssl-dev
    libssl-dev
    pandoc
    libxml2-dev
  • 可以看到，第一行是安裝OS
  • 第二行的RUN就在安裝R所需的system dependencies
  • 第三個區塊是安裝R package所需的system dependencies(這邊放入安裝tidyverse需要的dependencies)

3.3 R

  • 在Dockerfile中，安裝R，有2種方式：
    • 在base image的時候，就直接引入有R的image了(最常用，最簡單)
      # 從 RStudio的r-docker來的image
FROM rstudio/r-base:3.5-xenial
    • base image只有OS，然後我自己在這個OS下，用RUN的指令，去安裝R
      • 我可以用 wget 的方式，從給定的網址下載下來後，再安裝
        # download a version of R and build from source
ARG R_VERSION=3.5.2
RUN wget https://cdn.rstudio.com/r/ubuntu-1604/pkgs/r-${R_VERSION}_1_amd64.deb
RUN apt-get install -y gdebi-core
RUN gdebi r-${R_VERSION}_1_amd64.deb
      • 我也可以用OS內建的 package manager(e.g. centos的yum install, debian/ubuntu 的 apt-get install, macos的homebrew...)來下載安裝
        # not the recommended approach
# be sure you request a specific version of R
RUN apt-get install -y \
  r-base=3.4.4-1ubuntu1

3.4 R packages

  • 有兩類方式：
    • 把 R package也直接包到 image 裡面。
    • R package沒有放在 image裡面。而是等打開container去run base image時，在這個環境下，才去安裝 R package

3.4.1 把R package直接包到 imgae 裡面

  • 舉例來說，我可以在Dockerfile裡面，加上下面的指令，來安裝我要的 package:
    RUN R -e 'install.packages(c("tidyverse","lubridate"), repo = "https://rpkgs.company.com/frozen/repo/123")
  • 另一種作法是，我要他按照我的 renv.lock 文件，來安裝需要的package。這種作法我們就要先安裝 renv 這個 package，然後把 renv.lock 的文件從本機copy來container，然後才用 renv::restore() 去安裝。
    # 先安裝 renv 這個 package
ENV RENV_VERSION 0.13.2
RUN R -e "install.packages('remotes', repos = c(CRAN = 'https://cloud.r-project.org'))"
RUN R -e "remotes::install_github('rstudio/renv@${RENV_VERSION}')"
# 再把container的工作目錄設定好(optional)
WORKDIR /project
# 再把本機上的 renv.lock，複製一份到container空間上的工作目錄下
COPY local/renv.loc, renv.lock
# 再啟動 renv::restore()的功能去安裝 renv.lock 裡面列的 package
RUN R -e "renv::restore()"
  • 這樣做的好處是，我的 image 裡面就已經包好我要用的 package，所以要部署到任何地方都很方便，可以 container 開起來就直接用
  • 但這樣做的壞處是
    • 你的 image 可能很肥大，完全發揮不了 renv 的 global cache 優點。
    • 這個 image 太特定 for 這個專案使用，generalization能力極差( 因為別的專案，要用的package會和這個image不同，他還要自己update或remove package，那就會很難把你的這個 image 來當作他的 base image來用)
    • 還有一個明顯的挑戰是，裝這些 R package都要跟著安裝 linux 的 system dependency，你可能每寫入一個 R package 並 build image時，他就報 error，然後你就要照著這個error去補需要的 linux dependency 到 Dockerfile裡面，這常常會搞死人。那RStudio有賣個產品叫 RStudio Package Manager ，可以幫你解決這個問題，有興趣可以看這裡 (連結) ，但我們應該是不會碰到付費版的，所以也沒用

3.4.2 R package放在 image外，run container時才安裝

  • 最簡單的做法，就是開container時，做volume link，在本機放個package資料夾，讓他對應container空間中的package資料夾。那在container中安裝的package，就都可以被備份到本機中。那就不怕如果container斷掉或重啟時，原本裝好的package都不見。舉例來說，我的image是用 rocker/r-base，那我想裝 tidyverse 這個 package，我可以這樣做：
    #本機上的備份資料夾，用來備份container安裝的package
local_backup='r_package_backup'
#container上，R安裝的package都是裝到這邊。可用 .libPaths() 來查詢
container_lib_path='/usr/local/lib/R/site-library' 
sudo docker run --rm \
		-v ${local_backup}:${container_lib_path}
		R -e 'install.packages("tidyverse")'

Best Practices

目前看起來，最暴力但最簡單的做法是：

  • 寫一個 Dockerfile，裡面是以 rocker/rstudio + shiny-server 的 image 做基底，然後再灌 renv 這個package，再包成 image。就叫 hank/Rdev image 好了
  • 在 run hank/Rdev 這個 image時，要 mount 以下這堆東西
    • 在本機開個資料夾(就當作是本機專案的資料夾，e.g. web_of_dg 資料夾)，然後把他 mount 到 container 空間的 srv/shiny-server/web_of_dg。
    • 把本機的 var/log 和 container 的 log 給連結在一起
  • 要做 port 對應，假設是 -p 3456:3838
  • 進到RStudio Server後， setwd("srv/shiny-server/web_of_dg") ，就好像開啟 web_of_dg 這個專案一樣。
  • 在 web_of_dg 裡面，就像之前在測試區server的專案裡面一樣了，我要灌package，我就用 renv::install() ，那他就會灌到 web_of_dg/renv 裡面，然後因為 web_of_dg 這個資料夾和我的本機資料夾連動，所以我灌的 package 也都會保留在本機的 web_of_dg/renv 裡面
  • 如果開發過程中，安裝 R package，發現有 linux dependency的問題，需要安裝對應的 linux package，那就要重寫 hank/Renv 這個 image，在 RUN 的地方把這些 linux package給補上。
  • 接著，我寫的 code ，都透過 git 連到 gitlab 上
  • 我可以直接連到 172.16.99.233:3456/web_of_dg ，那就會幫我導到 container 裡的 3838/web_of_dg，然後啟動這個服務
  • 等到開發到一個段落了，要上線到正式區，這時就要到正式區，pull檔案下來，開container，run hank/Rdev 這個 image，並執行 renv::restore()，以及 port 對應。那就完成部署。
  • 覺得最後這步實在太麻煩，且沒辦法自動化，再想想怎做。

  • 等到開發到一個段落了，我要部署了，我就要重寫一個 Dockerfile 來建立部署用 image:
    • FROM rocker/shiny (以這個為基底)
    • RUN
      • linux dependency 的部分，把之前紀錄安裝過哪些linux package的檔案抓出來，複製貼上
      • (可能不用) R package 的部分，就直接 copy 本機的 renv.lock 到 Dockerfile，要他安裝這些。
    • COPY: 把本機的 web_of_dg 資料夾copy一份到 /srv/shiny-server 下面
    • 那這個 Dockerfile 執行後的 image ，應該就可以跑我的 R code
  • 開啟這個新image時，把剛剛本機的 web_of_dg ， mount 到 container 上，並設好 port 對應(e.g. 3456:3838)
  • 那只要連上 172.16.99.233:3456/web_of_dg ，就會導到 container 的 3838/web_of_dg，然後，這個資料夾裡 ui.R, server.R ，所以就會自動開啟 shiny 網頁。而這個網頁在跑時，有用到的 package，也都存在在 web_of_dg/renv/library 裡面
